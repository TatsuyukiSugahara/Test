# マルチスレッド設計ドキュメント

> 対象コミット: `b52a933` / 最終更新: 2026-07-03

対象: `aqEngine/Util/`(ThreadPool / Profiler)、`aqEngine/Rendering/RenderThread.*`、`aqEngine/Core/`(スレッド駆動)、`aqEngine/ECS/System.*`(並列 System)、`aqEngine/Resource/`(非同期ロード)

エンジンのスレッド構成は **ゲームスレッド + レンダースレッド + ワーカースレッドプール** の 3 種。フレームパイプラインと ECS の wave 並列、非同期リソースロードがこの上に乗る。

---

## 0. スレッド構成の全体像

```
┌──────────────── メイン(ゲーム)スレッド ────────────────┐
│ Application::Update()                                   │
│   Input → EntityContext::Update() → Resource::Update    │
│                    │  System を wave 並列で ThreadPool へ│
│                    ▼                                     │
│   Application::Render()  → RenderCommandList 構築        │
│                    │  RenderThread::Submit()             │
│   Application::FlushRender() → 前フレーム完了待ち         │
└─────────────────────┬───────────────────────────────────┘
                      │ Submit(list, lighting, shadow)
                      ▼
        ┌──────── レンダースレッド(RenderThread) ────────┐
        │ Run(): リングから取り出し RenderCommandList::   │
        │        Execute(RenderContext) → Present         │
        └─────────────────────────────────────────────────┘

        ┌──────── ワーカースレッドプール(ThreadPool) ────┐
        │ ECS System の wave 並列実行                      │
        │ ResourceManager の非同期ロード                  │
        └─────────────────────────────────────────────────┘
```

**3 つの並列軸**:
1. **フレームパイプライン**(ゲーム ⇄ レンダー) … `RenderThread` が担う。CPU(ゲーム構築)と GPU(描画実行)を 1 フレーム重複させる。
2. **ジョブ並列**(ECS System / リソースロード) … `ThreadPool` が担う。
3. **プロファイリング** … `Profiler` がスレッド別にロックフリー計測。

---

## 1. スレッドプール / ジョブシステム(`Util/`)

| ファイル | 責務 |
|---|---|
| `ThreadPool.h` / `.cpp` | **唯一の汎用スレッドプール(シングルトン)**。`Submit(f, args...)` がタスクを `std::packaged_task` 化してキューに積み、`std::future<戻り値型>` を返す。ワーカーは条件変数で待機し、順にタスクを実行。`Initialize(threadCount=0)` は 0 で論理コア数(`hardware_concurrency`)を使う。**ECS System の wave 並列**(`System.cpp`)と**リソース非同期ロード**の実行基盤。 |

**規約**: 並列処理は `ThreadPool` に一本化(`b52a933`)。かつて併存していた
`JobSystem`(依存 DAG 用)は実使用ゼロのデッドコードだったため撤去済み。
将来ジョブ間依存 DAG が必要になった場合は、`ThreadPool` の `std::future` チェーン
で表現するか、必要になった時点で専用機構を再設計する(凍結復活はしない)。

## 2. レンダースレッド(フレームパイプライン)

| ファイル | 責務 |
|---|---|
| `Rendering/RenderThread.h` / `.cpp` | **専用スレッドで `RenderCommandList` を実行するリングバッファ**。`Submit(list, displayRT, lighting, shadow)` でフレームスロットを充填し、`Run()`(ワーカーループ)が取り出して `Execute` → `CopyToBackBuffer` → `Present`。**フレーム単位の累計カウンタ**(`submittedCount_`/`completedCount_`/`prevFrameMark_`)で同期し、1 フレームに複数 Submit(オフスクリーン+メイン)しても正しく「前フレーム完了」を待てる。 |

**2 つの同期モード**(`RenderConfig.h` の `AQ_RENDER_PIPELINED`):

| モード | リング | 同期 | 効果 |
|---|---|---|---|
| **直列**(既定) | 2 スロット | `WaitForCompletion()` … 毎フレーム全完了を待つ | 実装が単純・レイテンシ最小 |
| **非同期**(PIPELINED) | 4 スロット | `WaitForPipelinedFrame()` … 前フレームの完了だけ待つ | フレーム N(ゲーム)と N-1(レンダー)が **1 フレーム重複** = スループット向上 |

- `FrameSlot` は per-draw/material/bones/ocean の `ConstantBufferPool` と lighting/shadow CB を各スロット独立で持つ(フレーム跨ぎのデータ競合回避)。
- **前提**: 非同期時はフレーム N と N+1 で別メイン RT を使う(Engine 側で毎フレームトグル)。
- DX12 移行: `Execute` を `ExecuteCommandLists` + フェンス完了通知に置換。リング/カウンタ機構はそのまま流用可能(設計コメントに明記)。

**スレッド安全の要**: DX11 immediate context は単一スレッド専有。ゲームスレッドは描画 API を一切呼ばず、`RenderFrame`(全 GPU リソースを `shared_ptr` 保持)を `Submit` で渡すことで、レンダースレッドが 1 フレーム遅れで安全に消費する。

## 3. ECS の wave 並列(`ECS/System.cpp`)

- `SystemManager::BuildSchedule()` … 依存グラフを Kahn 法でトポロジカルソートし、各 System に **level**(実行波)を割り当て。循環依存は assert。
- `SystemManager::Update()` … **level(wave)ごとに全 System を `ThreadPool::Submit` で並列投入**し、wave の全 future を `get()` で待ってから次 wave へ。例外は全完了後に最初の 1 つを rethrow。
- **競合回避**: System 中の構造変更(生成/破棄/Add/Remove)は ECS の**遅延コマンド**に積まれ、`FlushCommands()`(wave 完了後)で一括適用。並列実行中に Chunk 配置が動かない。

詳細は [ECS設計.md](04_ECS設計.md) §4。

## 4. 非同期リソースロード(`Resource/`)

詳細は [リソース管理設計](07_リソース管理設計.md) を一次資料とする。

**スレッド境界の契約(本書の管轄)**: デコード(CPU)はワーカースレッド。
GPU リソースの生成・アップロードは**メインスレッドの `ResourceManager::Update()` に集約**する。
レンダースレッドはコンテキスト操作(描画コマンド実行)のみを行い、リソース生成はしない。
DX11 のデバイスはフリースレッドだがこの規約で統一し、DX12 移行時はこの集約点を
コピーキュー化の候補とする。

## 5. プロファイラ(`Util/Profiler.*`)

| ファイル | 責務 |
|---|---|
| `Util/Profiler.h` / `.cpp` | **階層スコープ プロファイラ(Meyers シングルトン)**。`AQ_PROFILE_SCOPE(name)`/`AQ_PROFILE_FUNCTION()` マクロ(`ScopedTimer` RAII)。スコープの入れ子を親子階層で記録。**各スレッドが自身の `samples` にロックフリーで書き込み**、フレーム境界で `display` へ publish。メインは `PublishThisThread` + `PublishWorkers`(アイドルワーカー一括)、レンダースレッドは `MarkSelfPublishing` で自己 publish。非同期パイプライン時は `history` で「同周期に走ったフレーム」を選んで表示。`QueryPerformanceCounter` 基準。 |
| `Util/Debug/ProfilerDebugPanel.*` | スレッド別タイムラインの ImGui 表示。 |
| `Util/GameTimer.h` / `.cpp` | **フレームタイマー**。`QueryPerformanceCounter` 高精度計測。`Tick()` でデルタ更新(上限 `1/15秒` クランプ)+ FPS 制限のスピンウェイト。`GetDeltaTime`/`GetTotalTime`/`GetFPS`。 |

**計測のスレッド安全**: 各スレッドは自分の `ThreadData::samples` のみ操作(ロック不要)。`display`/`history` への publish 時だけ `publishMutex`。リリース(`AQ_DEBUG_IMGUI` 非定義)ではマクロが `((void)0)` に消える。

## 7. 1 フレームのスレッド連携(タイムライン)

```
メイン: [Input][ECS Update ─ wave並列 → ThreadPool ─][Resource Update][Render 構築]
                    │                                                    │ Submit
                    │                                     ┌──────────────┘
レンダー:           │        [ Execute フレームN-1 → Present ]（非同期時は重複）
                    │
ワーカー:    [Sys A][Sys B]      [decode job]  ...（ThreadPool が使い回し）

メイン: [FlushRender] ← 直列: フレームN完了待ち / 非同期: フレームN-1完了待ち
```

## 8. レビュー時のチェックポイント

- [ ] **描画 API のスレッド閉じ込め**: ゲーム/ワーカースレッドが D3D コンテキストを触っていないか。描画は `RenderThread` 単一に閉じているか。
- [ ] **フレーム重複の正当性**: 非同期パイプライン時、メイン RT トグル・`FrameSlot` 独立 CB プールでフレーム N/N-1 のデータが混ざらないか。累計カウンタ同期が複数 Submit で崩れないか。
- [ ] **遅延コマンドの徹底**: 並列 System 中の構造変更が即時実行されず FlushCommands に回るか(競合の根本回避)。
- [ ] **プリミティブの単一化**: 並列プリミティブが `ThreadPool` に一本化されているか(`JobSystem` は撤去済み)。新規並列コードが `ThreadPool` を使っているか。
- [ ] **プロファイラのロックフリー性**: スレッドが自領域のみ書き、publish 時のみロックする不変が守られているか。
- [ ] **停止時の安全**: 各プール/レンダースレッドの Finalize で、in-flight タスク/フレームを描き切ってからスレッドを畳むか(`Application::Finalize` は `FlushRender` 後に `renderThread_.Finalize`)。
- [ ] **future の例外伝播**: wave 実行で例外が全 task 完了後に rethrow され、ワーカーが中断しないか。
