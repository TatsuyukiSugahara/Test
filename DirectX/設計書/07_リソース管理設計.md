# リソース管理 設計ドキュメント

> 対象コミット: `700699e` / 最終更新: 2026-07-03

対象: `aqEngine/Resource/`、各リソース型のローダ(`Sound/SoundClip`、`Graphics` メッシュ/テクスチャ経路 等)
関連: [マルチスレッド設計](05_マルチスレッド設計.md)(ThreadPool)、[サウンド設計](03_サウンド設計.md)、[レンダリング設計](01_レンダリング設計.md)、`Platform/PlatformBudget.h`

非同期ロード・共有キャッシュ・重複排除・ローダ登録を担うリソース基盤。
スレッド境界の契約(デコード=ワーカー / GPU 反映=メインの `Update()` 集約)は**本書が一次資料**。

> **実装メモ**: キャッシュ(`Resource.h` の `TResourceBank`)は **世代を持たない共有キャッシュ**。
> `Unload<T>()` / `UnloadUnused()`(`8f16111`)でキャッシュから解放できる。世代カウンタは
> shared_ptr 所有モデルのため不採用(dangling が原理的に起きない。§3)。本書は実装に合わせて記述する。

## 0. 設計の全体像

```
Game / ECS
   │  ResourceManager::Get().Load<T>(path)   // 即座に shared_ptr<T> を返す
   ▼
ResourceManager（シングルトン）
   ├─ TResourceBank<T>::Find(key)            // 正規化パスで照会 → 既存なら共有 = 重複排除
   ├─ 新規なら loader = CreateLoader<T>()（Reflection 登録の工場）
   │        loader->StartAsync()             // ThreadPool::Get().Submit(...)
   │            └─ [ワーカースレッド] Loading()：ファイル IO + デコード（CPU のみ）
   └─ Update()（毎フレーム・メインスレッド）
            └─ ClearFinishedLoaders() → loader->Finish()
                 └─ FinalizeLoading()：GPU リソース生成（Texture/Shader）
                      → SetState(Completed / Invalid)
```

実コード対応:
- 入口: `ResourceManager::Load<Resource>(const char* path)`(`Resource.h`)。
- 非同期起動: `ResourceLoaderBase::StartAsync()` が `util::ThreadPool::Get().Submit([this]{ Loading(); ... })`(`Resource.cpp`)。
- メイン反映: `ResourceManager::Update()` → `ClearFinishedLoaders()` → `ResourceLoaderBase::Finish()` → `FinalizeLoading()`。

## 2. ロード状態機械

`ResourceBase` は `atomic<ResourceState>` で 3 状態を持つ(`Resource.h`):

```cpp
enum class ResourceState : uint8_t { Invalid, Loading, Completed };
// ctor で Loading に初期化。IsLoading()/IsCompleted()/IsFailed() が atomic を読む。
```

状態遷移(**状態変数は `ResourceState` のみ。デコード完了は別途 loader の `future_` で判定**):

```
                Load<T>()               Update()→Finish()
[生成: Loading] ───────▶ [Loading のまま] ───────────────▶ [Completed]  (IsCompleted()==true)
   │  (ctor)      worker: Loading() 実行       succeeded==true で FinalizeLoading() 成功
   │              (状態は変えず                       │
   │               loadSucceeded_ のみ更新)           └─ false ──▶ [Invalid]  (IsFailed()==true)
   │
   └─ Loading() が false / 例外 ─(loadSucceeded_=false)─▶ Update()→Finish() で [Invalid]
```

重要な事実(実コード準拠):
- **ワーカースレッドは `ResourceState` を変えない。** `Loading()` は `loadSucceeded_`(loader 内の `atomic<bool>`)だけを更新する。よって「デコード中」も「デコード完了・GPU 未反映」も、`ResourceBase` から見ると**どちらも `Loading`**。両者の区別は `ResourceLoaderBase::IsFinished()`(= `future_.wait_for(0)==ready`)で行う。
- **状態を `Completed`/`Invalid` に確定させるのはメインスレッドの `Finish()` のみ**(`Update()` 経由)。`IsCompleted()` が true を返すのは、`FinalizeLoading()` まで完了した後。
- **失敗(ファイル欠落・デコードエラー・例外)** … `Loading()` が false を返す/例外送出 → `loadSucceeded_=false` → `Finish()` で `SetState(Invalid)`。`IsFailed()`(旧綴り `IsFaild()` も alias)で検出。無限「ロード中」にはならない(次フレームの `Update()` で必ず確定する)。

## 3. キャッシュ・重複排除(世代管理は無し)

`TResourceBank<T>`(`Resource.h`)= 型別の `unordered_map<std::string, shared_ptr<T>>` + `mutex`。

- **パス正規化(キャッシュキー)**: `ResourceManager::NormalizeResourcePath()` が `\` を `/` に置換し、先頭 `./` を除去する。**大文字小文字の畳み込みはしない**(大小異なるパスは別キー)。キーは `GetContentRoot` を含まない生の要求パスなので、Win32/UWP で同一要求パスなら同一キーになる(実ファイル解決は §5 のローダ内で別途行う)。
- **重複排除**: `Load<T>` はまず `bank->Find(key)`(mutex 下)で既存を返す。無ければリソースを生成し `bank->RegisterOrGet(key, ref)` で登録。競合で既に登録済みなら**生成した loader を破棄して既存を返す**(`registerResult.second==false`)。→ 同一パスの同時 `Load` は**単一デコード**に合流する。
- **失敗エントリの遅延排除**: `Find`/`RegisterOrGet` は `IsFailed()` のエントリを見つけると map から erase する。→ 一度失敗したパスは次の `Load` で**再試行**される。
- **解放(unload)**: `ResourceManager::Unload<T>(path)`(単体除去)と `UnloadUnused()`(全型で `use_count()==1` = bank のみ参照の未使用リソースを解放)を提供する(`8f16111`)。バンクは通常 `shared_ptr` を保持し続けるが、これらでキャッシュから外してメモリを解放できる(シーン切替時など)。**ロード中・利用中**(ローダの `resource_` や呼び出し側が shared_ptr を保持)は `use_count>=2` のため `UnloadUnused()` では**誤解放しない**。`Unload<T>()` は強制除去だが、既存保持者があっても寿命は shared_ptr が管理するためクラッシュしない(bank が参照を手放すだけ)。
  - **世代カウンタは不採用**: 呼び出し側が `shared_ptr` で所有するモデルでは、bank が参照を手放しても既存保持者の寿命は shared_ptr が管理するため dangling が原理的に起きない。よって世代/ハンドル方式は導入していない(導入すると全呼び出し側のハンドル化という大改修になり、かつ安全上の追加価値が無い)。将来 raw ポインタ主体のハンドルアクセスへ移行する場合が導入のトリガー。

## 4. リソース型とローダ登録

ローダは **リソース型ごとに 1 つ**(`Reflection<T, Loader>()` で `Resource::ID()`= 型名 CRC32 をキーに登録)。**拡張子分岐はローダ内部**で行う。

| リソース型 | ローダ | デコード内容(worker `Loading()`) | GPU 反映(`FinalizeLoading()`) |
|---|---|---|---|
| `GPUResource`(テクスチャ) | `TextureLoader` | DDS=`LoadFromDDSFile` / その他=`LoadFromWICFile`、mip 生成、PCM 相当画素を `TextureData` へ | あり: `GraphicsDevice::CreateTexture2D`(**メイン**) |
| `ShaderResource` | `ShaderLoader` | `ParseShaderResourceKey`(記述子解析のみ) | あり: `GraphicsDevice::CreateShader`(**メイン**) |
| `MeshResource` | `MeshLoader` | `.tkm` を `LoadTkmMeshFile`(頂点/インデックス/マテリアルパス)、法線再計算 | なし(VB/IB は `StaticMesh::Initialize` で生成) |
| `SkeletalMeshResource` | `SkeletalMeshLoader` | `.tkm`(スキン頂点)+ 兄弟 `.tks`(ボーン) | なし(VB/IB は `SkeletalMesh` 側) |
| `AnimationResource` | `AnimationLoader` | `.tka` を `LoadTkaAnimationFile`(キーフレーム) | なし(CPU 常駐) |
| `SoundClip` | `SoundClipLoader` | 拡張子で WAV/MF 分岐 → PCM 全展開 | なし(CPU 常駐) |
| `PMDResource` | `PMDLoader` | `.pmd` ヘッダ読み(現状スタブ的) | なし |
| `FontResource`(UI) | `FontLoader` | フォントアセット | (UI 側) |

- 拡張子→処理の解決は各ローダの `Loading()` 内(例: `MeshLoader` は `GetLowerExtension()` で `.tkm` を判定)。新フォーマット追加は「新ローダを書く」か「既存ローダに拡張子分岐を足す」。
- 追加手順: ①リソース型 `T : res::ResourceBase`(`engineResource(名前)` マクロで `ID()` 付与) ②`Loader : res::ResourceLoaderBase` ③`RegisterBank<T,...>()` + `Reflection<T, Loader>()`(`Game/Application/Application.cpp` の `OnRegister`)。
- `FbxLoader` は登録経路にあるが `Loading()` は現状 `return true` のスタブ。

## 5. スレッド境界の契約(一次資料)

- **デコード(CPU)** … ワーカースレッド(`ThreadPool`)。`Loading()` でファイル IO・展開・パースまで。
- **GPU リソース生成・アップロード** … メインスレッドに集約する。実装上は 2 経路あるが**どちらもメイン/ゲームスレッド**で、ワーカー/レンダースレッドでは行わない:
  - テクスチャ/シェーダ: `ResourceManager::Update()` → `Finish()` → `FinalizeLoading()`(`CreateTexture2D`/`CreateShader`)。
  - メッシュ: `StaticMesh` / `SkeletalMesh` の `Initialize()`(ゲームスレッド)で VB/IB 生成。ローダの `FinalizeLoading` は使わない。
- **レンダースレッド** … コンテキスト操作(描画コマンド実行)のみ。リソース生成はしない。
- DX11 のデバイスはフリースレッドだが**この規約で統一**する。DX12 移行時はこの集約点をコピーキュー化の候補とする。
- ワーカーで COM/MF を使うデコーダ(`MFDecoder`)は自前で COM/MF を初期化する([サウンド設計](03_サウンド設計.md) §1)。

> **A-2 契約と実装の一致確認(2026-07-03 / `144ae8a`)**: `Resource.cpp` を確認した結果、
> GPU 反映は上記のとおり**メインスレッドに集約されており、契約文と実装は一致**する。
> ワーカーは `Loading()` の CPU デコードまで、`ResourceState` 確定と GPU 生成はメインの `Update()`/`Initialize()` 側。
> 唯一の注意点は「メッシュの GPU 反映がローダの `FinalizeLoading` でなく `StaticMesh::Initialize` にある」点で、
> スレッド(メイン/ゲーム)としては契約を満たす。

## 6. プラットフォーム予算との関係

`Platform/PlatformBudget.h` の `ResourceBudget`(コンパイル時定数)がリソース系に効く箇所:

| 予算項目 | Win32 | Xbox UWP(Game 分類) | 消費箇所 |
|---|---|---|---|
| `threadPoolWorkerCount` | 0(論理コア数) | 6 | `Engine::Initialize` の `ThreadPool::Initialize(...)`(デコード並列度の上限) |
| `stackSizeBytes` | 4MB | 8MB | `MemoryManager` 初期化(リソース確保の一部が乗る) |
| `memoryBudgetBytes` | 0(無制限) | 5GB | **観測(`700699e`)**。ヒープは OS 管理で非強制のため、`MemoryTracker` のライブ確保バイト総量を `MemoryManager::GetTrackedBytes()` / `IsOverMemoryBudget()` で観測できる(`_DEBUG` は実数・Release は 0/false)。強制はしない(設計方針どおり)。**Profiler パネルに使用量/予算バー + 確保サイト別の内訳を表示(`a1eb349`、[05 §5](05_マルチスレッド設計.md))** |
| `maxSingleFileBytes` | 0(制限なし) | 2GB | **強制(`700699e`)**。ロード時に `res::CheckFileBudget()` でファイルサイズを照合し、超過は診断ログ + ロード失敗で拒否。配線先: TKM/TKS/TKA(`OpenBinaryReadWithFallback` 経由)/ TKM メッシュ / PMD / テクスチャ(`file_size` best-effort)/ SoundClip。Win32 は 0 のため常に通過(実効は UWP のみ) |

## 7. レビュー時のチェックポイント

- [ ] **スレッド境界**: GPU リソース生成がワーカー/レンダースレッドに漏れていないか。テクスチャ/シェーダの `FinalizeLoading` がメインの `Update()` 経由か、メッシュ VB/IB がゲームスレッドの `Initialize` か。
- [ ] **解放・寿命**: `UnloadUnused()` / `Unload<T>()` が用意された。シーン切替時などに呼ぶ運用になっているか(呼ばないとキャッシュはアプリ寿命まで残る)。`use_count` ガードで利用中リソースを誤解放しないか。
- [ ] **重複排除**: 同一パスの同時 `Load` が単一 loader に合流するか(`RegisterOrGet` の二重防止)。二重デコードしていないか。
- [ ] **失敗経路**: ファイル欠落/デコード失敗が無限「ロード中」にならず、`Invalid` へ確定し次 `Load` で再試行されるか。
- [ ] **パス正規化**: Win32 / UWP(`GetContentRoot` 差)で同一アセットが同一キャッシュキーになるか。大小文字非畳み込みで別キー化していないか。
- [ ] **予算の実効性**: `maxSingleFileBytes` 超過ファイルが `res::CheckFileBudget()` で拒否されるか(UWP)。全ローダ経路(TKM/TKS/TKA/PMD/テクスチャ/SoundClip)に配線されているか。`memoryBudgetBytes` の観測(`IsOverMemoryBudget`)が `_DEBUG` で実数を返すか。`threadPoolWorkerCount` が実際に反映されているか。
- [ ] **Finalize 順序**: 終了時 `~ResourceManager` が in-flight loader を `Wait()`+`Finish()` してから破棄しているか。`ThreadPool::Finalize` との前後関係(ThreadPool がまだ生きているうちに待つ)。
