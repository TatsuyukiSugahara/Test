# git コミットメッセージ規約

このリポジトリのコミット件名(1行目)は必ず **`<タグ> 内容`** の形にする。
タグは山括弧 `<>` で囲み、直後に**半角スペース1つ**を置いてから内容を書く。

```
<Engine> シェーダの実行時コンパイルをワーカースレッドへ移しロードヒッチを解消
<Game> タイトル画面(案A「墨」)を実装: 筆文字「残刃」+朱の落款+入退場アニメ
```

- 内容は**日本語**で、何を・なぜ変えたかが分かる要約にする(既存履歴に合わせる)。
- 件名の末尾に句点は付けない。詳細は必要なら空行を挟んで本文へ。

---

## 1. タグ一覧(この4つを使う)

タグは**変更した「領域」を第一基準**に選ぶ。種別(修正/追加)より領域を優先する。

| タグ | 使うとき |
|---|---|
| `<Game>` | **ゲーム側 / アプリ側**を変えたとき(`Game/...`、ゲームプレイ、UI、タイトル/ローディング等の画面、ゲーム固有ロジック) |
| `<Engine>` | **エンジン処理**を変えたとき(`aqEngine/...`、レンダリング、リソース管理、Level/ECS/シーン、サウンド基盤など汎用機構) |
| `<Build>` | プロジェクト/ビルド設定(`.vcxproj` / `.filters` / `.props`、フィルター整合、依存関係、ビルド構成) |
| `<Chore>` | 上記に当てはまらない雑務(ドキュメント、コメント整理、`.gitignore`、Claude スキル、細かな片付け) |

---

## 2. タグの決め方(領域優先)

**「どこを変えたか」を最優先**に判断する。バグ修正・機能追加・リファクタといった
**種別ではタグを分けない**。

- エンジンのバグ修正 → `<Engine>`(`<Fix>` にはしない)
- ゲーム側の新機能 → `<Game>`
- エンジンのリファクタ → `<Engine>`

判断の目安:

1. 変更ファイルが `aqEngine/...` 主体 → **`<Engine>`**
2. 変更ファイルが `Game/...` 主体 → **`<Game>`**
3. 変更が `.vcxproj` / `.filters` / `.props` など**ビルド構成中心** → **`<Build>`**
   (ソース追加に伴う登録の同期は、その機能側のタグ = Game/Engine に含めてよい。
    プロジェクト整理そのものが主目的のときだけ `<Build>`。)
4. どれでもない雑務 → **`<Chore>`**

### 領域がまたがるとき

1コミットが Game と Engine の両方に触れる場合は、**主目的の領域**でタグを付ける。
本来は領域ごとにコミットを分けるのが望ましい。分けられないときは、
主となる側のタグを選び、本文でもう一方への波及を書き添える。

---

## 3. コミット手順

1. `git status` / `git diff` で変更範囲を確認し、主目的の領域を判定する。
2. §1 の表からタグを1つ選ぶ。
3. 件名を `<タグ> 内容` で作る(スペース1つ、句点なし、日本語)。
4. コミットする。Claude Code の既定フッタ(`Co-Authored-By: ...`)は
    ユーザー方針に従い付与する。

```bash
git commit -m "<Engine> メッシュクラスタの遅延生成をスレッドセーフ化"
```

---

## 4. 注意点

- **タグは必ず1つ・先頭に置く**。`<Engine><Game>` のような連結はしない。
- 山括弧の**直後は半角スペース1つ**。全角スペースや詰め書きにしない。
- 迷ったら「**このコミットで一番変えたのはゲームか、エンジンか**」に立ち返る。
  種別(Fix/Feat/Refactor)で迷う必要はない ─ 領域で決める。
- 過去履歴には `<Fix>` `<Feat>` `<Refactor>` `<App>` 等の別タグも残っているが、
  **今後は Game / Engine / Build / Chore の4つに集約**する。
