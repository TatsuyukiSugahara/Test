# カリング 設計ドキュメント

対象: `aqEngine/Rendering/`, `aqEngine/Graphics/`, `aqEngine/Component/`, `aqEngine/Resource/`
前提: 既存の抽象レイヤ（`RenderContext` / `IRenderContextImpl`）を変更せず、**DX11 / DX12 の両方**でカリングを有効化する。
本書は **設計のみ**。実装コードは含まない。

対象カリング:
1. **トライアングルカリング**（クラスタ／メッシュレット単位）
2. **フラスタムカリング**
3. **オクリュージョンカリング**（Hi-Z 方式）

採用方針（確定事項）:
- **CPU 駆動から段階的に**実装し、後から GPU 駆動（Indirect Draw）へ拡張する。
- トライアングルカリングの粒度は **クラスタ（メッシュレット）単位**。
- DX11 / DX12 で**コードを二重化しない**。差分は最終フェーズの間接描画 API にのみ閉じ込める。

---

## 0. 設計の指針

- **抽象レイヤ（`RenderContext` / `IRenderContextImpl`）はできる限り変更しない。** フラスタム／オクリュージョン（CPU リードバック方式）は抽象レイヤを触らずに導入できる。間接描画が必要になる最終フェーズでのみ `RenderContext` に API を追加する。
- カリングロジックは可能な限り **API 非依存の層**（`BuildRenderFrame` 〜 `BuildCommandList`、および compute シェーダ）に置く。これにより DX11 / DX12 が自動的に共有する。
- 各フェーズは単独で効果が出て、後戻りせず積み上がる構成とする（§7 フェーズ計画）。
- メインカメラのカリングと**シャドウ（ライト視錐台）のカリングを混同しない**。画面外でも影は落ちるため、判定基準を分ける（§3.3）。

---

## 1. 現状アーキテクチャと挿入ポイント

```
[ゲームスレッド]
  RenderSystem::BuildRenderFrame()      ... ECS を走査し RenderItem を frame へ push
        ↓  frame.items (deferred) / frame.forwardItems
  Renderer::BuildCommandList()          ... RenderItem → RenderCommandList
        ↓
[レンダースレッド]
  RenderCommandList::Execute(RenderContext&)   ... DX11/DX12 共通の高レベル API
        └─ impl_ : IRenderContextImpl
              ├─ D3D11RenderContextImpl
              └─ D3D12RenderContextImpl
```

| 層 | 場所 | カリングでの役割 |
|---|---|---|
| バウンディング生成 | `Resource::MeshResource`（`Resource.h`） | ロード時にローカル AABB / 境界球 / クラスタを計算・保持 |
| アイテム構築 | `RenderItem`（`Rendering/RenderFrame.h`） | ワールド変換前のバウンディングを保持 |
| **フラスタムカリング** | `RenderSystem::BuildRenderFrame()`（`Component/BodyComponentSystem.cpp`） | push 直前に視錐台内外判定（CPU・共通） |
| HZB 生成 / オクリュージョンテスト | `Renderer::BuildCommandList()` + 新規 compute コマンド | GBuffer 深度から Hi-Z を作り、AABB をテスト |
| オクリュージョン結果反映 | 次フレームの `BuildRenderFrame()`（CPU リードバック） | 前フレームの可視フラグで除外 |
| **トライアングル（クラスタ）カリング** | 新規 compute パス + 間接描画 | クラスタ単位で可視判定し compact、Indirect Draw |

**最重要**: `RenderItem` は `worldMatrix` を持つがバウンディング情報を持たない。`MeshResource` は頂点／インデックスを持つが AABB を計算していない。よって §2 の共通基盤がすべての前提となる。

## 2. 共通基盤: バウンディングデータ

### 2.1 メッシュ側（ロード時に一度だけ計算）

`MeshResource`（および `SkeletalMesh` / `Terrain` / `HeightmapChunk` の各メッシュ供給元）に、ロード完了時に以下を計算・キャッシュする:

| データ | 内容 | 用途 |
|---|---|---|
| ローカル AABB | `center`, `extent`（半サイズ） | フラスタム／オクリュージョン |
| 境界球 | `center`, `radius` | 高速な早期判定（任意） |

- `FbxLoader` / `MeshLoader` の `FinalizeLoading()` 内、または頂点配列確定後に min/max を走査して算出する。
- スケルタルメッシュは**バインドポーズ AABB を少し膨らませる**（アニメで頂点が境界を超える分のマージン）。厳密にやるならボーン OBB の合成だが、初期はマージン付き AABB で十分。

### 2.2 RenderItem 側

`RenderItem` に以下を追加し、各 `FillRenderItem()` で埋める:

```
math::Vector3 boundsCenter;   // ローカル AABB 中心
math::Vector3 boundsExtent;   // ローカル AABB 半サイズ
// 境界球を使うなら float boundsRadius;
```

`worldMatrix` は既存。カリング時にワールド AABB を導出する（§3.1）。

### 2.3 ワールド AABB 導出（共通ヘルパ）

ローカル AABB（center, extent）にワールド行列 `M` を適用する標準式（回転を吸収して軸並行 AABB を再構成）:

```
worldCenter = M * localCenter
worldExtent.x = |M00|*ext.x + |M01|*ext.y + |M02|*ext.z
worldExtent.y = |M10|*ext.x + |M11|*ext.y + |M12|*ext.z
worldExtent.z = |M20|*ext.x + |M21|*ext.y + |M22|*ext.z
```

`Math` に `AABB` 構造体と上記変換ヘルパを新規追加する（`Vector.h` に `Dot`/`Cross` 等は既存）。

## 3. フラスタムカリング（Phase 1・CPU・DX11/DX12 共通）

### 3.1 フラスタム抽出

`Math` に `Frustum`（6 平面）を新規追加。`Camera::GetViewProjectionMatrix()` から Gribb–Hartmann 法で 6 平面（left/right/bottom/top/near/far）を抽出する。各平面は `(nx, ny, nz, d)` で正規化保持。

### 3.2 判定と挿入

`RenderSystem::BuildRenderFrame()` の各 `Foreach<...>` 内、`FillRenderItem()` 成功後・`push_back` 直前に:

1. ローカル AABB → ワールド AABB（§2.3）
2. ワールド AABB × 6 平面の内外判定（球で早期 reject → AABB で確定、の二段が高速）
3. 完全に外なら **push しない**

これだけで `frame.items` / `forwardItems` が視錐台内に絞られ、GBuffer・forward・後続オクリュージョン処理すべてが軽くなる。

### 3.3 シャドウとの分離（必須）

- メインカメラのフラスタムで切ってよいのは「画面に直接映る描画」のみ。
- `castShadow == true` のアイテムは、メインカメラで外でも**ライト視錐台に入れば影として必要**。
- よって `ShadowPass`（`Rendering/Shadow/`）は**ライトの視錐台**で別途フラスタムカリングする。`BuildRenderFrame` で main 用に弾いたアイテムをシャドウから誤って除外しないよう、シャドウ用の収集は別経路 or 別フラグで管理する。

### 3.4 DX11/DX12 差

なし。完全に CPU・API 非依存。

---

## 4. オクリュージョンカリング（Phase 2・Hi-Z・CPU リードバック）

ハードウェアオクルージョンクエリ方式は GPU ストール・API 差・1 フレーム遅延が大きく**非採用**。
既に `RenderContext` に `Dispatch` / CS UAV・SRV があるため **Hi-Z（HZB）方式**を採る。

### 4.1 データフロー（フレーム N）

```
フレーム N-1 の深度（GBuffer0 の depth）
        │
        ▼ (compute) ミップ連鎖生成: 各レベル = 親 2x2 の "最遠（max）" 深度
   Hi-Z バッファ（HZB, mip chain）
        │
        ▼ (compute) 各オブジェクトのワールド AABB をスクリーン投影し、
        │            覆う矩形に対応する HZB ミップで深度比較
   可視フラグ配列（UAV, オブジェクトごとに 0/1）
        │
        ▼ CPU へリードバック（1 フレーム遅延）
   フレーム N+1 の BuildRenderFrame で不可視オブジェクトを除外
```

### 4.2 HZB 生成

- 入力: deferred の GBuffer 深度（`DeferredRenderer::GetGBuffer0Handle()` 近傍、`Renderer::BuildCommandList()` Pass 2 の後）。
- compute で max-reduction のミップ連鎖を生成。新規 RT/UAV テクスチャ（R32_Float）を確保。
- DX11/DX12 とも `Dispatch` + CSUAV で共通実装。

### 4.3 オクリュージョンテスト

- 入力: フラスタム通過済みオブジェクトのワールド AABB 配列（structured buffer）+ HZB + view-proj。
- 各 AABB をクリップ空間に投影 → スクリーン矩形と最近接深度を算出 → 矩形サイズに応じた HZB ミップをサンプルし、AABB 最近接深度 ≤ HZB 値なら可視。
- 出力: 可視フラグ UAV。

### 4.4 結果反映（CPU リードバック方式・本フェーズ）

- 可視フラグを CPU にリードバックし、**オブジェクト ID と対応付けて**保持。
- 次フレームの `BuildRenderFrame` で、前フレーム不可視だったオブジェクトを push しない。
- **1 フレーム遅延**するため、カメラ急変時の「一瞬の抜け」を許容するか、保守的判定（迷ったら可視扱い）でポップを防ぐ。
- オブジェクトと結果を対応付ける安定 ID が必要（ECS Entity ID を使う）。

### 4.5 DX11/DX12 差

- HZB 生成・テストの compute は共通。
- リードバック（GPU→CPU コピー）は `IRenderContextImpl` 経由。DX11 は staging、DX12 は readback ヒープ + フェンスだが、**いずれも既存の抽象に薄く追加**で済む（新規描画 API は不要）。

---

## 5. トライアングル（クラスタ）カリング（Phase 4・GPU 駆動）

CPU で全三角形を毎フレーム回すのは非現実的。ラスタライザのバックフェースカリングは GPU が常時実施済み。よって**クラスタ（メッシュレット）単位の GPU カリング + 間接描画**を採る。本フェーズは間接描画基盤（§6）が前提。

### 5.1 前処理（ロード時）

- メッシュを ~64〜128 三角形のクラスタに分割（meshoptimizer 等のアルゴリズム、または単純な連続分割）。
- 各クラスタに **境界（AABB/球）** と **法線錐（cone: 軸 + 角度）** を計算・保持。

### 5.2 ランタイム（compute）

各クラスタに対し:
1. フラスタムカリング（クラスタ境界 × 視錐台）
2. バックフェースカリング（法線錐 × 視線方向）
3. Hi-Z オクリュージョン（§4 の HZB を流用）

通過したクラスタのインデックス範囲を compact し、間接描画引数バッファ（draw count / index 範囲）を構築。

### 5.3 描画

- 構築した間接引数で `DrawIndexedInstancedIndirect`（DX11）/ `ExecuteIndirect`（DX12）。
- 将来 DX12 は Mesh Shader 化も可能だが、本設計の範囲外。

---

## 6. 間接描画基盤（Phase 3・API 差をここに閉じ込める）

オクリュージョンの「遅延なし GPU 駆動化」とクラスタカリングの両方が、これを前提とする。

`RenderContext` / `IRenderContextImpl` に間接描画 API を追加:

| 抽象 API（新規） | DX11 実装 | DX12 実装 |
|---|---|---|
| `DrawIndexedIndirect(argBuffer, offset)` | `DrawIndexedInstancedIndirect` | `ExecuteIndirect`（1 引数の command signature） |
| 間接引数バッファ生成 | UAV 付き Buffer | UAV 付き Buffer + command signature |

これにより、オクリュージョン結果反映を §4.4（CPU リードバック）から **GPU 駆動 2-phase occlusion**（前フレーム可視物 → HZB → 今フレーム再テスト、遅延なし）へ置き換えられる。

---

## 7. フェーズ計画

| Phase | 内容 | API 差 | 抽象レイヤ変更 |
|---|---|---|---|
| **0** | バウンディング基盤: `MeshResource` AABB 計算、`RenderItem` 拡張、`Math::AABB`/`Frustum` 追加 | なし | なし |
| **1** | フラスタムカリング（CPU・共通）＋ シャドウのライト視錐台カリング | なし | なし |
| **2** | Hi-Z オクリュージョン（HZB 生成 + テスト compute、CPU リードバック反映） | リードバックのみ薄く追加 | 最小限（readback） |
| **3** | 間接描画 API（`DrawIndexedIndirect`）を抽象に追加 → オクリュージョンを GPU 駆動 2-phase 化 | ここに集約 | あり（間接描画） |
| **4** | クラスタ（メッシュレット）カリング = トライアングルカリング | Phase 3 を流用 | なし（Phase 3 で完了） |

各フェーズは独立して効果が出る。Phase 0→1 だけでも実用上の描画負荷は大きく下がる。

- `RenderingDebugPanel`（`Rendering/RenderingDebugPanel.*`）に各カリングの **ON/OFF トグル**と **可視/総数カウンタ**を追加。

**CPU 方式クラスタカリングの性能特性（重要）**: 毎フレーム「128クラスタの可視判定 + 可視インデックスの compact + 動的IBアップロード」を行うため、**小メッシュ（~8千三角形のキャラ等、GPU が三角形律速でない）では CPU/転送コスト > GPU 節約で差し引きマイナス**。実機計測で確認済み。よって既定 OFF。巨大メッシュ / GPU 律速シーンでのみ手動 ON する道具とする。本質的に速くするには §11（GPU 駆動）が必要。

## 11. B 案メモ: トライアングル（クラスタ）カリングの GPU 駆動化

目的: CPU 方式の「毎フレーム compact + IB アップロード + ゲーム/レンダー二重計算」を排し、**GPU 上でクラスタを判定 → 可視クラスタだけを `ExecuteIndirect` で描画**する。CPU は判定にもインデックス転送にも関与しない。

### 11.1 必要な新規インフラ（抽象レイヤ）

1. **構造化バッファ / 生バッファ UAV**（現状 `IStructuredBuffer` は空スタブ）
   - クラスタ記述子 SRV（入力: center/extent/coneAxis/coneCutoff/triOffset/triCount）
   - 可視クラスタ index 出力 UAV（append 風 / カウンタ付き）
   - 間接引数バッファ UAV（`D3D12_DRAW_INDEXED_ARGUMENTS` 列）
2. **間接描画 API**: `RenderContext::DrawIndexedIndirect(argBuffer, offset)`
   - DX12: command signature（`D3D12_INDIRECT_ARGUMENT_TYPE_DRAW_INDEXED`）+ `ExecuteIndirect`
   - DX11: `DrawIndexedInstancedIndirect`
3. **UAV カウンタのクリア**（`ClearUnorderedAccessViewUint` 相当）

### 11.2 ランタイム フロー

```
[ロード時] メッシュ → クラスタ記述子バッファ (SRV) + 並べ替え済みインデックスバッファ (IB)
                       ※ Meshlet.cpp の出力をそのまま GPU へ常駐

[毎フレーム・compute]
  ClusterCull.cs:
    入力 SRV: clusters, viewProj/camPos CB, (任意) Hi-Z
    各クラスタ並列:
      - フラスタム（クラスタ AABB × 視錐台）
      - バックフェース（法線錐 × 視線）
      - (任意) Hi-Z オクリュージョン（§4 を流用）
      可視なら:
        appendIdx = InterlockedAdd(visibleCounter, 1)
        visibleClusterList[appendIdx] = clusterIndex
    その後 1 スレッドで:
        indirectArgs.IndexCountPerInstance = Σ可視クラスタ.triCount*3
        ... または「クラスタ単位の ExecuteIndirect 多重ドロー」にする

[描画]
  方式A: 可視クラスタの三角形を別 compute で compactedIB へ書き出し → 単一 DrawIndexedIndirect
  方式B: クラスタごとに 1 ドロー（多重 ExecuteIndirect, MaxCommandCount=クラスタ数）
         → compaction 不要だがドローコール増。最近の GPU では方式 B も実用的。
```

推奨は **方式A（compute で compactedIB を作り単一間接描画）**。Hi-Z は既存の `HiZRenderer` をクラスタ判定にも供給すれば、オブジェクト単位とクラスタ単位の遮蔽を統合できる。

### 11.3 既存資産の流用

- クラスタ記述子は `graphics::MeshCluster`（`Meshlet.h`）をそのまま GPU レイアウト化（16B 境界に整列して `float4` 詰め）。
- 並べ替え済みインデックスは `GetReorderedIndices()` を GPU 常駐 IB にする（CPU 方式の `cullIndexBuffer_` は不要化）。
- 判定式（フラスタム/バックフェース錐）は `IsClusterVisible`（`Meshlet.h`）を HLSL へ移植。`coneCutoff` の意味（`dot(viewDir, coneAxis) >= coneCutoff` で背面）も同じ。
- Hi-Z は `HiZRenderer` の R32 ピラミッドを SRV で供給。

### 11.4 実装フェーズ（段階導入・各段ビルド可能）

1. 抽象に `IStructuredBuffer`（SRV/UAV/カウンタ）+ `DrawIndexedIndirect` を追加（D3D12 実装）。クリア画面で UAV 書き込み→間接描画の最小往復を検証。
2. クラスタ記述子 + 並べ替え IB を GPU 常駐化。`ClusterCull.cs`（フラスタムのみ）で compactedIB 生成 → 単一間接描画。CPU 方式と同じ「画面端で削減」を GPU で再現。
3. バックフェース錐 + Hi-Z をクラスタ判定に追加。
4. CPU 方式（`ClusterCull.cpp` / `cullIndexBuffer_`）を撤去 or デバッグ用に残置。

### 11.5 リスク / 注意

- D3D12 の `ExecuteIndirect` + command signature + UAV カウンタ + バリアは新規実装で**実機検証必須**（CPU では追えない）。Step 2 のリードバック同様、診断カウンタを仕込みながら段階確認する。
- DX11 と DX12 で間接引数バッファ/シグネチャの作法が異なる。差は `IRenderContextImpl` 実装に閉じ込める。
- 小メッシュでは GPU 駆動でも「ドロー準備の固定コスト」が効果を上回ることがある。巨大メッシュ / 高ポリシーンで初めて実効。

## 12. テライン / 海をクラスタカリング対象に含める（メモ）

現状クラスタを持つのは `MeshResource`（StaticMesh）/ `SkeletalMeshResource`（SkeletalMesh）経由のメッシュのみ。**テライン・海は未対応**（集計にも実カリングにも入らない）。

### 12.1 テライン（`Terrain/HeightmapChunk`）

- 頂点は `vertCache_`（CPU 常駐）にある。`MeshResource` を介さず `InitializeDynamic` で動的VB生成しているためクラスタ未生成。
- 対応案:
  1. `HeightmapChunk` に `std::vector<graphics::MeshCluster> clusters_` + `reorderedIndices_` を追加し、`BuildMesh` / `RebuildFromHeights` 後に `graphics::GenerateClusters(vertCache の position 列, indices, 64, ...)` で生成（AABB と同様にキャッシュ・dirty 管理）。
  2. `FillRenderItem` で `item.clusters / reorderedIndices / cullIndexBuffer` を設定（StaticMesh と同じ要領）。動的IBは別途用意。
- 効果: テラインは**広く平らで上向き**のため、バックフェースはほぼ効かない。効くのは**フラスタム（チャンクが画面外/部分外）**。地形を複数チャンク化すると更にフラスタムが効く。**地形は「チャンク AABB のオブジェクト単位フラスタム」で先に切る方が安く効く**ので、クラスタ化は優先度低め。

### 12.2 海（`Ocean/OceanMesh`）

- `OceanMesh` は StaticMesh/SkeletalMesh ではなく独自。Gerstner 波で**毎フレーム頂点が動く**ため、クラスタの AABB / 法線錐が毎フレーム変わる → ロード時固定クラスタは不正確。
- 対応案:
  - 海は通常**カメラ中心の巨大グリッド**で全面可視のことが多く、クラスタカリングの効果は小さい。優先度低。
  - やるなら「波の最大振幅でクラスタ AABB を膨らませた保守的クラスタ」をロード時生成し、フラスタムのみ適用（バックフェースは波で不安定なため無効化）。

### 12.3 まとめ（優先度）

| 対象 | クラスタ化の価値 | 推奨 |
|---|---|---|
| テライン | フラスタムのみ（チャンク単位で十分なことが多い） | まず**チャンク AABB のオブジェクト単位フラスタム**を入れる。クラスタ化は後 |
| 海 | 低（全面可視・頂点が動く） | 当面見送り。やるなら保守 AABB + フラスタムのみ |

いずれも GPU 駆動（§11）が入れば、頂点が動く海も compute 側で動的にクラスタ境界を作れるため相性が良くなる。
