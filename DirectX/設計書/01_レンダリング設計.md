## 0. 設計の全体像

レンダリングは **2 つのレイヤー**に分かれる。

| レイヤー | 名前空間 | フォルダ | 役割 |
|---|---|---|---|
| **グラフィックス抽象 (低レベル)** | `aq::graphics` | `Graphics/` | GPU API(D3D11/D3D12/Vulkan)を **Bridge パターン**で抽象化。リソース生成・ステート設定・ドローの発行だけを担う。API 固有型を上位に漏らさない。 |
| **レンダリング (高レベル)** | `aq::rendering` | `Rendering/` | 「1 フレームに何を描くか」を組み立てる。`RenderFrame`(スナップショット)→ `RenderCommandList`(記録)→ 専用スレッドで再生。ディファード/シャドウ/ポストプロセス/オクリュージョンの各パスを持つ。 |

ハイブリッド(DX11/DX12/Xbox/UWP)の要は **この 2 層の境界**にある。上位の `aq::rendering` は API を一切知らず、切り替えは `Graphics/` 内の実装差し替えだけで済む。

```
[ゲームスレッド]
  ECS(RenderSystem::BuildRenderFrame) → RenderFrame（描画スナップショット）
        │  フラスタムカリング済み
        ▼
  Renderer::BuildCommandList()  → RenderCommandList（API 非依存コマンド列）
        │  Submit()
        ▼
[レンダースレッド]  RenderThread
  RenderCommandList::Execute(RenderContext&)
        └─ RenderContext → impl_ : IRenderContextImpl
              ├─ D3D11RenderContextImpl
              ├─ D3D12RenderContextImpl
              └─ VulkanRenderContextImpl
```

## 1. グラフィックス抽象レイヤー(`aqEngine/Graphics/`)

### 1.1 Bridge パターンの中核

| ファイル | 責務 |
|---|---|
| `GraphicsDevice.h/.cpp` | **抽象デバイス(シングルトン)**。`Create<TImpl>()` テンプレートでバックエンド実装を注入する。`Initialize`/`Finalize`/`Present`、リソースファクトリ(`CreateVertexBuffer` 等)、オフスクリーン RT 生成、`OnSuspend/OnResume`(UWP PLM)を公開。内部に `IGraphicsDeviceImpl` を 1 つ持ち、全呼び出しを委譲する。上位コードはこのクラスしか触らない。 |
| `IGraphicsDeviceImpl.h` | **デバイス実装インターフェース**(Bridge の Implementor)。D3D11/D3D12/Vulkan の各実装が継承。デバイス/スワップチェーン初期化、リソースファクトリ、`ReadbackOffscreenR32`(GPU→CPU リードバック)、GPU 駆動バッファ生成(`CreateStructuredBuffer`/`CreateRawBuffer`)を宣言。未対応バックエンド用に安全な既定実装(no-op/nullptr)を持つ。 |
| `RenderContext.h` | **抽象コンテキスト**(コマンド発行の窓口)。`OMSet*`/`RSSet*`/`IASet*`/`VS/PS/CSSet*`/`Draw`/`Dispatch`/`DrawIndexedIndirect` 等の描画コマンドを、すべて**抽象インターフェース型引数**で受ける。`IRenderContextImpl` に委譲するだけの薄いラッパ。 |
| `IRenderContextImpl.h` | **コンテキスト実装インターフェース**。`DepthMode`/`BlendMode` 列挙もここで定義。GPU 駆動間接描画(`IASetIndexBufferGpu`/`DrawIndexedIndirect`/`UavBarrier`)とスライス指定シャドウは既定 no-op で、対応バックエンドのみ override する。 |
| `RenderContext.cpp` | `Execute` 系ヘルパ・`RenderContext` の非インライン実装。 |

**レビュー観点**: `RenderContext` は全メソッドがインラインで `impl_->` に流すだけ。API 追加時は「抽象 IF(no-op 既定)→ 各実装 override」の順で足す規約になっているか(間接描画がこの流儀)。

| `IShader.h` | シェーダ抽象 IF。`ShaderType`(VS/PS/CS 等)を持つ。 |

### 1.3 バックエンド実装(D3D11 / D3D12 / Vulkan)

3 API が **同じファイル構成で 1:1 に並ぶ**。ビルド構成 `ENGINE_GRAPHICS_D3D11 / _D3D12 / _VULKAN` で 1 つだけコンパイルされる。

| D3D11 (`Graphics/D3D11/`) | D3D12 (`Graphics/D3D12/`) | Vulkan (`Graphics/Vulkan/`) | 役割 |
|---|---|---|---|
| `D3D11GraphicsDeviceImpl` | `D3D12GraphicsDeviceImpl` | `VulkanGraphicsDeviceImpl` | `IGraphicsDeviceImpl` 実装。デバイス/スワップチェーン/キュー初期化、リソースファクトリ、Present、リードバック。 |
| `D3D11RenderContextImpl` | `D3D12RenderContextImpl` | `VulkanRenderContextImpl` | `IRenderContextImpl` 実装。DX11=immediate context、DX12/Vulkan=コマンドリスト記録。 |
| `D3D11Buffers` | `D3D12Buffers` | `VulkanBuffers` | 頂点/インデックス/定数バッファ具象。 |
| `D3D11Shader` | `D3D12Shader` | `VulkanShader` | シェーダコンパイル/ロード。 |
| `D3D11Resources` / `D3D11RenderResources.h` | `D3D12Resources` / `D3D12RenderTarget` | `VulkanResources` / `VulkanRenderTarget` | SRV/サンプラー/UAV/RT の具象。 |
| `D3D11DepthMap` | `D3D12DepthMap` | `VulkanDepthMap` | シャドウ深度マップ具象。 |
| — | `D3D12RootSignature` | `VulkanPipelineLayout` | ルートシグネチャ/ディスクリプタレイアウト。DX11 には概念が無いため無し。 |
| — | `D3D12PipelineStateCache` | `VulkanPipelineCache` | PSO キャッシュ(DX12/Vulkan は PSO 明示のためキャッシュ必須。DX11 は動的ステートで不要)。 |
| — | `D3D12GpuBuffer` | — | GPU 駆動クラスタカリング用バッファ(構造化/RAW/間接引数)。 |
| `imgui_impl_dx11`(ThirdParty) | `D3D12ImGui` | `VulkanImGui` | ImGui バックエンド接続。 |
| — | `D3D12Common.h` | `VulkanCommon.h` / `VulkanVMAImpl.cpp` | 共通ヘルパ(DX12=ヘルパ、Vulkan=VMA アロケータ)。 |

**ハイブリッドの肝**:
- **DX11** … immediate context・動的ステート中心で最も単純。Xbox One UWP(FL11_0 未満 = App 分類の FL10_0 経路)フォールバックの主経路。compute/UAV/間接描画は FL によって非対応 → `SetComputeSupported(false)` で機能を落とす。具体的な FL 値の根拠は [Xbox移植設計](../aqEngine/Platform/Xbox移植設計.md) に一元化。
- **DX12** … 明示的な PSO/ルートシグネチャ/フェンス/リードバックヒープ。GPU 駆動カリング・Hi-Z リードバックが完全動作する主戦力。実機 Xbox のフル描画経路。
- **Vulkan** … 同一構造の先行事例(抽象 IF 追加 0 本で移植した実績)。SDK 未導入時は `#ifdef` ガードで実コンパイル対象外。

## 2. レンダリングレイヤー(`aqEngine/Rendering/`)

### 2.1 フレームデータとコマンド基盤

| ファイル | 責務 |
|---|---|
| `RenderFrame.h` | **1 フレームの描画スナップショット**。ゲームスレッドが構築しレンダースレッドが消費。`CameraData`/`RenderItem`(1 ドロー分。GPU リソースは全て `shared_ptr` で 1 フレーム遅延消費に耐える)/`OceanRenderItem`/`DecalRenderItem` を保持。`items`(deferred)/`forwardItems`(透明)/`oceanItems`/`decalItems` に分類。カリング用ローカル AABB・GPU 駆動カリングバッファもここ。 |
| `IRenderCommand.h` | **描画コマンド基底 IF**。`Execute(RenderContext&, FrameContext&)` の 1 メソッドのみ。 |
| `RenderCommandList.h/.cpp` | **コマンドを 64KB ページのリニアアリーナに placement-new して記録・再生**。`Enqueue<T>()` で積み、`Execute()` で順に再生、`Reset()` でデストラクタを呼びカーソルだけ戻す(次フレーム再利用)。ページは再確保しないので保持ポインタが常に有効。DX12 移行では `Execute` をコマンドリスト記録に置換予定。 |
| `FrameContext.h/.cpp` | **フレーム単位のリソースプール**。`ConstantBufferPool`(per-draw b0/material b2/bones b4/ocean b5 を draw ごとに Allocate、フレーム跨ぎで再利用)。`FrameContext` は全コマンドの `Execute` に渡され、CB スロットや確定 displayRT を供給。 |
| `RenderTargetHandle.h` | RT を指す軽量ハンドル(メイン RT / オフスクリーン RT を統一的に参照)。 |
| `RenderConfig.h` | `AQ_RENDER_PIPELINED`(直列/非同期切替)等のビルド設定。 |

### 2.2 レンダラー本体とスレッド

| ファイル | 責務 |
|---|---|
| `Renderer.h/.cpp` | **記録フェーズの司令塔**。`BuildCommandList(frame, outList, ...)` で `RenderFrame` を `RenderCommandList` に変換。シャドウ/ディファード/ポストプロセスの各レンダラを保持し、パス順序を組む。UI 描画と Hi-Z 構築は `std::function` コールバックで疎結合化(Renderer が UI/Occlusion に直接依存しない)。`RenderDebugSync`(_DEBUG 専用の同期実行)も持つ。 |
| `RenderThread.h/.cpp` | 専用スレッドでコマンドリストを実行するリングバッファ。同期モード・リング構成・累計カウンタによるフレーム同期の詳細は [マルチスレッド設計 §2](05_マルチスレッド設計.md) を一次資料とする。 |
| `RenderFrame.h`(前掲) | — |

**レビュー観点(スレッド)**: DX11 immediate context は単一スレッド専有。ゲームスレッドは絶対に描画 API を触らず、`Submit` 経由でレンダースレッドに渡す規約。`RenderDebugSync` は「レンダースレッドから呼べ」とコメントで縛っている。非同期パイプライン時は「フレーム N と N+1 で別メイン RT を使う」(Engine 側トグル)が前提。

## 3. フレームの流れ(`Application::Render` 実体)

`Core/Application.cpp` が毎フレーム行う手順(ハイブリッド分岐込み):

1. メインカメラ位置を `LightManager` に反映。
2. **ImGui**: `ImGui_ImplWin32_NewFrame` + バックエンド別 `NewFrame`(DX11/DX12/Vulkan を `#ifdef` 分岐)。FPS/バックエンド名/VSync(DX12)のオーバーレイ。
3. `OnPreRender()`(ゲーム固有のオフスクリーンパス Submit)。
4. メイン `RenderCommandList` を確保し、SetRenderTarget/Clear/SetViewport を Enqueue。
5. `RenderSystem::BuildRenderFrame(mainFrame)` … ECS 走査+**フラスタムカリング**で `RenderFrame` を構築。
6. `Renderer::BuildCommandList(...)` … シャドウ→G-Buffer→Hi-Z→ライティング→ポストプロセス→UI→ImGui の順にコマンド記録。
7. `RenderThread::Submit(...)` … レンダースレッドへ。表示 RT はポストプロセス最終出力(`GetDisplayRTHandle`)。

初期化時(`Application::Initialize`)の**機能ゲート**が重要:
- `IsComputeSupported()==false`(FL11_0 未満 = Xbox One UWP App 分類の FL10_0 経路 等)→ クラスタカリング無効・Hi-Z 未初期化。
- ディファードが有効なときだけ Hi-Z を初期化しオクリュージョンテスターに登録。

## 4. レビュー時のチェックポイント

- [ ] **抽象の漏れ**: `aq::rendering` 層や上位に D3D 型(`ID3D11*`/`ID3D12*`)が出ていないか。出るのは `Graphics/D3D1x/` 内のみが正。
- [ ] **API 追加の順序**: 新コマンドは「`IRenderContextImpl` に no-op 既定 → 対応バックエンドのみ override」で足せているか(未対応 API を壊さない)。
- [ ] **スレッド規約**: 描画 API 呼び出しがレンダースレッドに閉じているか。`shared_ptr` によるフレーム遅延消費の寿命が守られているか。
- [ ] **機能ゲート**: compute/UAV/間接描画/リードバックが `IsComputeSupported()` や各バックエンドの既定 no-op で安全にフォールバックするか(DX11 / FL11_0 未満経路)。
- [ ] **メモリ**: `RenderCommandList` の 64KB ページ・`ConstantBufferPool` の high-water mark 増加が意図通りか。
- [ ] **カリングの整合**: GPU/CPU 経路の min-clusters 閾値統一、メインカメラとシャドウのカリング分離。
- [ ] **RenderItem の shared_ptr コスト**: アイテム数増加時の per-item shared_ptr コピー(atomic 参照カウント)が許容範囲か。大量描画テスト時に計測すること。
